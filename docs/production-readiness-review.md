# 正式使用前生产就绪评估

评估日期：2026-05-09

评估目标：支撑约 1000 名学生和教师共同使用，重点关注数据安全、提交可靠性、并发写入、查询性能、部署运维和后续扩展。

## 总体结论

当前平台已经适合小规模真实试用和教学试点：账号、班级、作业、试卷、提交记录、备份和后台恢复都已具备基础能力。

如果要正式面向约 1000 人使用，P0 中的数据安全和核心运行数据拆表已经完成主要阶段：学生提交写入独立 SQLite `attempts` 表，登录会话写入独立 SQLite `sessions` 表，用户、班级、报名、作业和审计日志也已写入独立 SQLite 表；学习中心和班级学情已增加第一阶段分页。非 GET 写操作仍保留串行化保护，降低同时写入造成覆盖的风险。后续 P0 重点建议转为练习记录页、教师概览和后台用户列表分页，以及把试卷结构从整体 JSON 继续拆表。

建议上线策略：

1. 先完成 P0 必做项。
2. 再用 1 个班级做 50-100 人压测和恢复演练。
3. 通过后再扩大到全年级或全校使用。

## P0：正式大规模使用前必须完成

### 1. 提交记录结构化和分页

风险等级：最高

当前状态：

- 已新增独立 SQLite `attempts` 表，学生提交会写入独立表。
- 旧整体运行状态 JSON 中的 `attempts` 会自动迁移到独立表。
- 学习中心、班级学情、用户管理和教师概览已改为查询独立表。
- 学习中心已完成已完成作业和错题分页；班级学情已完成学生表和最近答题分页。
- 练习记录页、教师概览最近答题和后台用户列表仍需要补充分页参数。

风险：

- 提交写入覆盖风险已明显降低。
- 如果单个用户或教师概览长期积累大量提交记录，剩余未分页接口仍可能变慢。

建议改造：

- 练习记录页增加页码和每页数量。
- 教师概览最近答题、后台用户列表、学生列表继续增加后端分页。

验收标准：

- 1000 名用户各提交 5 次后，提交接口稳定返回。
- 任意学生历史记录、学习中心、班级学情加载时间可控。
- 连续并发提交不丢记录。

### 2. 核心运行数据拆表

风险等级：最高

当前状态：

- 用户、班级、报名、作业、审计日志已经拆入独立 SQLite 表。
- 整体 `db` JSON 不再保存用户、班级、报名、作业、审计日志、会话和提交记录数组。
- 非 GET 写操作已串行化，降低了并发写入覆盖风险。
- 登录会话已经拆入独立 SQLite `sessions` 表，登录、登出和过期清理为表级写入。

风险：

- 当前实现先通过 `saveDb` 将内存中的核心数组同步到独立表，业务行为稳定；后续可以继续把具体路由优化为单行 upsert，减少高峰期写入量。
- `paperVisibility` 和考试类型配置仍在轻量整体状态中。
- 试卷仍是整体 JSON 数组，编辑并发和题目级查询还需要继续优化。

建议改造：

- 将 `paper_visibility` 和考试类型配置继续拆为 SQLite 表。
- 将用户创建、班级创建、报名、发布作业和审计记录逐步从整表同步改为单行 upsert。
- 保留 `data/db.json` 仅作为初始种子或导出文件，不再作为运行状态。

验收标准：

- 创建用户、加入班级、发布作业、修改密码均持久化到独立表。
- 后台用户列表、学生列表和班级列表继续补充分页。
- 同时创建账号或发布作业不会互相覆盖。

### 3. 试卷结构和编辑并发保护

风险等级：高

当前状态：

- 试卷已写入 SQLite 的 `app_state.papers`，同时同步 `data/papers.json`。
- 试卷仍是整体 JSON 数组。

风险：

- 多个管理员或教师同时编辑试卷时，后保存的人可能覆盖先保存的内容。
- 试卷数量增多后，列表、搜索和导出会变慢。

建议改造：

- 短期：给试卷增加版本号或 `updatedAt` 并做保存冲突检测。
- 中期：拆分 `papers`、`questions`、`choices`、`answers` 表。
- 导入 Word 时先生成预览和差异确认，再写入。

验收标准：

- 两个用户同时编辑同一试卷时，后保存者能看到冲突提示。
- 试卷列表支持分页、类型筛选和关键词搜索的后端查询。

### 4. 恢复演练和离线备份

风险等级：高

当前状态：

- 已有自动备份、后台一键备份和后台恢复。
- 备份目录默认在本机 `backups/`。

风险：

- 服务器磁盘损坏或误删时，本机备份也会一起丢失。
- 没做恢复演练时，无法确认备份在真实事故中可用。

建议改造：

- 每周至少做一次恢复演练。
- 每天把 `backups/` 同步到另一台服务器或对象存储。
- 备份完成后记录文件大小、试卷数量、用户数量、提交数量。
- 恢复前后自动校验关键数量。

验收标准：

- 能在新目录或测试服务器中从备份恢复并登录。
- 恢复后试卷数、用户数、班级数、提交数和备份清单一致。

## P1：正式上线首周内强烈建议完成

### 1. 分页和后端筛选

影响范围：

- 管理试卷列表
- 学生列表
- 用户管理
- 班级学情
- 最近答题
- 学生历史记录
- 错题本

建议：

- 所有列表接口增加 `page`、`pageSize`、`keyword`、`classId`、`paperId`、`userId` 参数。
- 前端只渲染当前页。
- 默认每页 20-50 条，导出功能另走后台任务。

### 2. 静态资源版本化

当前风险：

- 更新后浏览器可能继续使用旧的 `app.js` / `styles.css`。
- 用户已经多次遇到“服务器更新了但页面没变”的情况。

建议：

- 在 `index.html` 引用中加入版本号，例如 `app.js?v=20260509`。
- 更进一步可加入构建步骤，用文件 hash 生成资源名。

### 3. 日志轮转和监控告警

建议：

- 安装 `pm2-logrotate`。
- 监控：
  - CPU
  - 内存
  - 磁盘空间
  - PM2 进程状态
  - 500 错误数量
  - 登录失败数量
  - 备份成功/失败状态

### 4. Nginx/HTTPS/安全头

建议：

- 强制 HTTPS。
- 设置 `client_max_body_size 20m`。
- 增加基础安全头：
  - `X-Content-Type-Options`
  - `Referrer-Policy`
  - `X-Frame-Options`
- 如果后续嵌入第三方资源，再单独设计 CSP。

### 5. 默认账号和权限运营流程

建议：

- 正式上线前删除或停用演示账号。
- 管理员账号使用强密码。
- 建立账号批量导入前检查流程。
- 记录账号创建、批量导入、角色变更和恢复操作。

## P2：稳定运行后逐步完善

### 1. 自动保存答题草稿

建议：

- 学生答题过程中每 20-30 秒保存草稿。
- 刷新或重新打开试卷时自动恢复草稿。
- 提交成功后清理草稿。

### 2. 数据归档

建议：

- 按学期或考试周期归档旧班级、旧作业和旧提交。
- 日常页面默认只查当前学期。
- 历史数据通过归档入口查询。

### 3. Word 导入导出任务化

建议：

- 大量试卷导出改成后台任务。
- 任务完成后显示下载入口。
- 防止大文件导出阻塞主进程。

### 4. 审计增强

建议记录：

- 试卷修改前后的摘要。
- 发布作业的目标班级。
- 批量导入成功/失败明细。
- 恢复备份的操作者、恢复目标和安全备份名称。

## P3：体验和扩展优化

- 教师工作台首页：展示班级、待查看学情、最近作业和快捷发布入口。
- 学生端移动适配：优化手机和平板答题体验。
- 考试模式：限时、自动交卷、提交后查看策略、禁止反复提交。
- 错题与知识点：按题型、等级、知识点统计薄弱项。
- 运行评测：如果开放编程题，必须接入沙箱、任务队列和资源限制。

## 推荐上线检查清单

正式开放前逐项确认：

- 已使用正式管理员账号，默认演示账号已停用或删除。
- HTTPS 已启用，HTTP 自动跳转 HTTPS。
- PM2 为单实例运行，未启用 cluster。
- `data/` 和 `backups/` 不纳入 Git 提交。
- 后台一键备份可用。
- 备份恢复已在测试目录演练过。
- 至少完成 100 人同时登录和提交测试。
- 服务器磁盘空间充足，并配置日志轮转。
- 更新流程已固定：备份 -> git pull -> npm install -> pm2 restart -> 强制刷新。

## 建议实施顺序

1. 试卷结构化入库和编辑冲突检测。
2. 练习记录页、教师概览和后台列表分页。
3. 核心写入从整表同步逐步优化为单行 upsert。
4. 静态资源版本化。
5. 恢复演练和离线备份。
6. 日志轮转和监控。
7. 自动保存答题草稿。
8. 数据归档和导出任务化。
